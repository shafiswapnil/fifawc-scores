## What This Is

**WC Scores** — a native, lightweight macOS menu bar app showing FIFA World
Cup 2026 match schedules, live scores, and goal animations. Menu bar agent
(no Dock icon). All interaction happens in the menu bar — no main window.

**Current status:** Phase 1 complete (M1–M14). Phase 2 planned (M15–M18).
Read `status.md` for exact milestone state.

## Agent Behavior Rules (MANDATORY)

These are non-negotiable. The user has reinforced these across every session.

1. **Read `status.md` first.** Know the current stage before doing anything.
2. **Don't hallucinate.** If you're unsure, say so. Don't guess file contents,
   API responses, or error messages. Read the file first.
3. **Crosscheck before claiming done.** After building something, verify:
   - All files parse/compile (`swiftc -parse`)
   - All wiring is correct (model ↔ view ↔ service)
   - No duplicate keys, no unused imports, no silent bugs
4. **Ask before major decisions.** If something isn't specified in the docs,
   ask the user. Don't assume.
5. **Update docs after every change.** When you complete a milestone or make
   a key decision, update:
   - `status.md` (milestone tracker + decisions log)
   - `docs/plan.md` (if new milestone added)
   - `docs/notes.md` (if new decision or idea)
   - `prompts.md` (log the prompt + summary)
6. **Commit each milestone separately** with Conventional Commits format.
7. **Keep commits clean.** One logical change per commit. Never mix
   feature code with doc updates in the same commit.
8. **Energy and efficiency.** The user values fast, focused work.
   Do everything in one prompt when possible. Minimize back-and-forth.
9. **Never edit generated files.** `WCSCORES.xcodeproj` is generated by
   XcodeGen — edit `project.yml` instead.

## Build, Run, Test

```bash
# Regenerate the Xcode project after editing project.yml
xcodegen generate

# Quick syntax check (no Xcode needed)
swiftc -parse Sources/**/*.swift \
  -sdk /Library/Developer/CommandLineTools/SDKs/MacOSX.sdk \
  -target arm64-apple-macosx14.0

# Full build
xcodebuild -project WCSCORES.xcodeproj -scheme WCSCORES \
  -configuration Debug -destination 'platform=macOS' \
  CODE_SIGNING_ALLOWED=NO build
```

**Note:** This machine may not have Xcode installed. Use `swiftc -parse`
for syntax validation. Full builds require Xcode.

## Key Constraints (Non-Negotiable)

- **Tiny RAM/CPU footprint.** < 20 MB RAM, < 1% CPU during live polling.
- **Zero login items.** User opens manually; app stays in menu bar once launched.
- **No Dock icon.** `LSUIElement = true`. Menu bar agent only.
- **macOS 14+ deployment target.** Universal binary (Apple Silicon + Intel).
- **No settings window.** Everything in the menu bar panel.
- **All scheduling logic respects the chosen poll interval** (min 60s).
- **One allowed third-party dependency:** Sparkle (auto-update framework, ~2MB).
  Everything else is native (URLSession, SwiftUI, etc.).

---

## Conventions

- **Swift 6** with strict concurrency (`SWIFT_STRICT_CONCURRENCY=complete`).
- **SwiftUI-first**, minimal AppKit bridging only where SwiftUI gaps exist.
- **`@Observable`** for state management (no Combine publishers).
- **`@MainActor`** for MatchStore and PollController (clean concurrency).
- **`actor`** for FetchService (network state isolation).
- **`URLSession`** for networking (no third-party HTTP libraries).
- **UserDefaults** (`@AppStorage`) for persisted settings (API key, poll interval, favorite team).
- **Conventional Commits:** `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `test:`.
- **XcodeGen** for project generation (`project.yml` → `WCSCORES.xcodeproj`).
- **Semver** for releases: `MARKETING_VERSION` in `project.yml` MUST match the git tag.

## Architecture Quick Reference

```
WCScoresApp (@main)
  └─ @State store = MatchStore()  ← injected via .environment()
       ├─ FetchService (actor)     ← football-data.org API client
       ├─ PollController           ← task-based polling state machine
       ├─ matchesByDate            ← [String: [Match]] grouped by date
       ├─ standings                ← [GroupStanding] (groups A–H)
       └─ settings                 ← apiKey, pollInterval, favoriteTeam (UserDefaults)
            ├─ MenuBarLabel        ← compact: ⚽ WC / ⚽ BRA vs ARG · 3PM / ⚽ BRA 2-1 ARG · 67'
            └─ MenuBarPanel        ← 6 tabs: Today/Yesterday/Tomorrow/Schedule/Standings/Settings
                 ├─ MatchCard      ← team flags, score, status, group
                 ├─ GroupStandingCard ← group table with positions
                 └─ SettingsView   ← API key field, poll slider, favorite team picker
```

## Version & Release Rules

- `MARKETING_VERSION` in `project.yml` is the single source of version truth.
- Git tags must match: `v1.1.0` tag → `MARKETING_VERSION: "1.1.0"`.
- Sparkle compares `CFBundleShortVersionString` against GitHub release tags.
- Mismatch causes false "update available" prompts — always verify before tagging.
- Use `./scripts/bump-version.sh X.Y.Z` (M17) to enforce consistency.
