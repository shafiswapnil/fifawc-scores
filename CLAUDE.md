     After a **full phase** ends, also verify CLAUDE.md architecture section
     still matches reality (new files, changed wiring, etc.).
     Do NOT maintain a Git History section — just run `git log`.
6. **Update README.md after big milestones.** The public `README.md` must
   reflect the latest state of the project — new features, updated tech
   stack, updated architecture diagram, updated settings table, etc.
7. **Update README-PERSONAL.md for big milestones.** The personal
   `README-PERSONAL.md` (git-ignored) is the user's maintenance reference.
   When new workflows, tools, or release steps are added, update this file
   so the user understands what changed and how to use it.
8. **Commit each milestone separately** with Conventional Commits format.
9. **Keep commits clean.** One logical change per commit. Never mix
   feature code with doc updates in the same commit.
10. **Energy and efficiency.** The user values fast, focused work.
    Do everything in one prompt when possible. Minimize back-and-forth.
11. **Never edit generated files.** `FIFAWCSCORES.xcodeproj` is generated by
    XcodeGen — edit `project.yml` instead.

## Build, Run, Test

```bash
# Regenerate the Xcode project after editing project.yml
xcodegen generate

# Quick syntax check (no Xcode needed)
swiftc -parse Sources/**/*.swift \
  -sdk /Library/Developer/CommandLineTools/SDKs/MacOSX.sdk \
  -target arm64-apple-macosx14.0

# Full build
xcodebuild -project FIFAWCSCORES.xcodeproj -scheme FIFAWCSCORES \
  -configuration Debug -destination 'platform=macOS' \
  CODE_SIGNING_ALLOWED=NO build
```

**Note:** This machine may not have Xcode installed. Use `swiftc -parse`
for syntax validation. Full builds require Xcode.

### CI Runner Requirements

- **CI runs on `macos-15`** (Xcode 16+) — do NOT downgrade to `macos-14`
  (Xcode 15.4 can't read project format 77 generated by latest XcodeGen).
- **XcodeGen is installed via `brew install xcodegen`** in each CI job —
  it's NOT pre-installed on GitHub runners.
- **`swiftc -parse` is NOT used in CI** — it can't resolve SPM packages
  (like Sparkle). `xcodebuild` handles full compilation.
- **Node.js 20 deprecation warning** is cosmetic — safe to ignore until
  September 2026.

## Key Constraints (Non-Negotiable)

- **Tiny RAM/CPU footprint.** < 20 MB RAM, < 1% CPU during live polling.
- **Zero login items.** User opens manually; app stays in menu bar once launched.
- **No Dock icon.** `LSUIElement = true`. Menu bar agent only.
- **macOS 14+ deployment target.** Universal binary (Apple Silicon + Intel).
- **No settings window.** Everything in the menu bar panel.
- **All scheduling logic respects the chosen poll interval** (min 60s).
- **One allowed third-party dependency:** Sparkle (auto-update framework, ~2MB).
  Everything else is native (URLSession, SwiftUI, etc.).

## Conventions

- **Swift 6** with strict concurrency (`SWIFT_STRICT_CONCURRENCY=complete`).
- **SwiftUI-first**, minimal AppKit bridging only where SwiftUI gaps exist.
- **`@Observable`** for state management (no Combine publishers).
- **`@MainActor`** for MatchStore and PollController (clean concurrency).
- **`actor`** for FetchService (network state isolation).
- **`URLSession`** for networking (no third-party HTTP libraries).
- **UserDefaults** (`@AppStorage`) for persisted settings (API key, poll interval, favorite team).
- **Conventional Commits:** `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `test:`.
- **XcodeGen** for project generation (`project.yml` → `FIFAWCSCORES.xcodeproj`).
- **Semver** for releases: `MARKETING_VERSION` in `project.yml` MUST match the git tag.

### Swift 6 Concurrency Gotchas (MUST KNOW)

- **`@Observable` + `didSet` self-assignment = infinite recursion / crash.**
  Never write `self.property = newValue` inside a `didSet` of an `@Observable`
  property. Use a private backing variable with a computed get/set instead.
- **Actor-isolated calls need `await`.** Calling actor methods from
  `@MainActor` or non-isolated context requires `await`. Use
  `Task { await actor.method() }` in synchronous contexts like `didSet`.
- **`.environment()` is a View modifier, not a Scene modifier.** Apply it
  to child views inside `MenuBarExtra { } label: { }`, not to the scene itself.
- **`PRODUCT_MODULE_NAME`** must be set explicitly when `PRODUCT_NAME`
  differs from the import module name (tests `import FIFAWCSCORES`).
- **`TimelineView` hangs MenuBarExtra label.** Do not use `TimelineView` inside
  a `MenuBarExtra { } label: { }` closure — it freezes the label and the app
  becomes unresponsive. Use an `@Observable` property (e.g. `minuteTick: Date`)
  updated by a `Task { while true { try? await Task.sleep(...); self.minuteTick = .now } }`
  loop instead. MenuBarLabel observes the property and SwiftUI re-renders on change.

## Architecture Quick Reference

```
FIFAWCScoresApp (@main)
  ├─ SPUStandardUpdaterController (Sparkle) ← auto-update
  └─ @State store = MatchStore()  ← injected via .environment()
       ├─ FetchService (actor)     ← football-data.org API client
       ├─ PollController           ← task-based polling state machine
       ├─ matchesByDate            ← [String: [Match]] grouped by date
       ├─ standings                ← [GroupStanding] (groups A–H)
       └─ settings                 ← apiKey, pollInterval, favoriteTeam (UserDefaults)
            ├─ MenuBarLabel        ← 5 states: idle(⚽ FWC) / upcoming(⚽ ESP vs CPV · 10PM) / live(⚽ ESP 1-0 CPV · 67') / HT(⚽ ESP 1-0 CPV · HT) / finished(⚽ SWE 5-1 TUN · FT)
            |                       real-time tick via store.minuteTick (Task.sleep 60s loop)
            |                       GOAL!: text flips to "GOAL!" for 2s via triggerGoal()
            └─ MenuBarPanel        ← 6 tabs: Today/Yesterday/Tomorrow/Schedule/Standings/Settings
                 ├─ MatchCard      ← team flags, score, status, group
                 ├─ GroupStandingCard ← group table with positions
                 ├─ SettingsView   ← API key field, poll slider, favorite team picker
                 └─ "Check for Updates…" button → Sparkle
```

## Version & Release Rules

- `MARKETING_VERSION` in `project.yml` is the single source of version truth.
- Git tags must match: `v1.1.0` tag → `MARKETING_VERSION: "1.1.0"`.
- Sparkle compares `CFBundleShortVersionString` against GitHub release tags.
- Mismatch causes false "update available" prompts — always verify before tagging.
- Use `./scripts/bump-version.sh X.Y.Z` (M17) to enforce consistency.
